# Project 4 — Build a Crew and Govern It: Grading Rubric

| **Criteria** | **Novice** | **Competent** | **Proficient** |
|--------------|------------|---------------|----------------|
| **Design (Part 1)** | **Points: 8** — Workflow description is vague or generic; agent table is incomplete or gives every agent the same tools; orchestration choice is stated without justification; human checkpoint is missing or names an arbitrary step with no reasoning. | **Points: 14** — Workflow is specific and the agent table is complete, but the "tools it does NOT need" column is thin or absent; orchestration choice is justified but without reference to the workflow's specific characteristics; human checkpoint identifies a reasonable step but the approval packet or reversibility question is underdeveloped. | **Points: 20** — Workflow is concrete and real; agent table names both what each agent needs and what it explicitly does not get; orchestration choice is justified with specific reference to the workflow's structure; human checkpoint names the right step with a clear explanation, five approval-packet fields, and a direct answer on reversibility. |
| **Build (Part 2)** | **Points: 8** — Final canvas not meaningfully different from the AI Copilot's initial output; Process type does not match the Part 1c choice and no explanation is given; Human Input feature missing with no documentation; fewer than two screenshots present. | **Points: 14** — Canvas is adjusted from the Copilot's initial output and the Process type matches the Part 1c choice, but tool scoping adjustments are minor or not tied back to Part 1b; Human Input is enabled or limitation is noted but without explanation; most screenshots present. | **Points: 20** — Final canvas clearly reflects the Part 1 design; Process type matches the Part 1c choice; tool adjustments trace back to specific Part 1b scoping decisions; Human Input is enabled on the correct task (or limitation is documented with what would be configured and why); all three screenshots present and legible. |
| **Run and Timeline (Part 3)** | **Points: 6** — Run used placeholder or minimal input; handoff observations are missing or cover only one agent; human checkpoint experience not addressed. | **Points: 11** — Run used realistic input; handoff observations cover two agents but stay surface-level (output noted but not evaluated for correctness); human checkpoint experience addressed but without evaluating whether it delivered the Part 1d approval packet. | **Points: 15** — Run used realistic, specific input; handoff observations cover at least two agents and assess whether each output was correct for the next agent; if the Human Input checkpoint fired, the submission evaluates whether the pause gave the reviewer what Part 1d said they would need; execution timeline screenshot shows at least two agents' outputs. |
| **Governance Audit (Part 4)** | **Points: 12** — Design vs. reality section missing or acknowledges no changes; tool scope table absent or not compared to the Part 1b plan; threat class table is generic and not tied to this crew; audit trail evaluation answers fewer than two of the three questions; top 3 risks and mitigations are vague or missing. | **Points: 21** — Design vs. reality section identifies some changes but does not explain all of them; tool scope table is mostly complete but discrepancies are unexplained; threat class table addresses the crew but mitigations are stated without specificity; audit trail evaluation answers all three questions but one or more stays general; risks and mitigations are named but not specific to this crew or not prioritized. | **Points: 30** — Design vs. reality section honestly compares the Copilot's output to the final crew and explains every change; tool scope table is complete and discrepancies are explained; threat class table is specific to this crew with concrete mitigations (not aspirational statements); audit trail evaluation answers all three questions with specifics from this build; top 3 risks and mitigations are crew-specific and ordered by priority. |
| **Reflection (Part 5)** | **Points: 6** — One or both questions missing, or answered with general statements not tied to this build. | **Points: 11** — Both questions answered, but Question 1 describes the governance decision without explaining the reasoning behind it, or Question 2 names a general role ("IT," "the developer") rather than a specific function. | **Points: 15** — Question 1 names a specific governance decision made during the build and explains the judgment behind it (not just what was done, but why); Question 2 names an actual organizational function (not "IT" or "the engineer") and identifies the specific timeline entry that function would need to investigate the failure. |
