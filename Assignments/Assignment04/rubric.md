# Project 4 — Build a Crew and Govern It: Grading Rubric

| **Criteria** | **Novice** | **Competent** | **Proficient** |
|--------------|------------|---------------|----------------|
| **Design (Part 1)** | **Points: 8** — Workflow description is vague or generic; agent table is incomplete or gives every agent the same tools; orchestration choice is stated without justification; human checkpoint is missing or names an arbitrary step with no reasoning. | **Points: 14** — Workflow is specific and the agent table is complete, but the "tools it does NOT need" column is thin or absent; orchestration choice is justified but without reference to the workflow's specific characteristics; human checkpoint identifies a reasonable step but the approval packet or reversibility question is underdeveloped. | **Points: 20** — Workflow is concrete and real; agent table names both what each agent needs and what it explicitly does not get; orchestration choice is justified with specific reference to the workflow's structure; human checkpoint names the right step with a clear explanation, five approval-packet fields, and a direct answer on reversibility. |
| **Build (Part 2)** | **Points: 8** — Final canvas not meaningfully different from the AI Copilot's initial output; Process Type does not match the Part 1c choice and no explanation is given; human-checkpoint task is missing or its Description and Guardrail are not configured; fewer than two screenshots present. | **Points: 14** — Canvas is adjusted from the Copilot's initial output and the Process Type matches the Part 1c choice, but tool scoping adjustments are minor or not tied back to Part 1b; human-checkpoint task exists and its Description references waiting for approval, but the Guardrail is generic or absent; most screenshots present. | **Points: 20** — Final canvas clearly reflects the Part 1 design; Process Type matches the Part 1c choice; tool adjustments trace back to specific Part 1b scoping decisions; human-checkpoint task is in place with a Description that specifies waiting for explicit human confirmation AND a Guardrail that requires concrete approval evidence (named approver, timestamp, or similar); all three screenshots present and legible. |
| **Run and Timeline (Part 3)** | **Points: 6** — Run used placeholder or minimal input; handoff observations are missing or cover only one agent; the human-checkpoint behavior is not addressed. | **Points: 11** — Run used realistic input; handoff observations cover two agents but stay surface-level (output noted but not evaluated for correctness); human-checkpoint task is mentioned but the submission does not say whether a real human was involved or what the agent actually produced. | **Points: 15** — Run used realistic, specific input; handoff observations cover at least two agents and assess whether each output was correct for the next agent; the human-checkpoint behavior is documented honestly — including the duration of that task, whether any real human input occurred, and whether the agent's "approval" output names a real approver or a placeholder; Timeline screenshot shows at least two agents' completed task entries. |
| **Governance Audit (Part 4)** | **Points: 12** — Design vs. reality section missing or acknowledges no changes; tool scope table absent or not compared to the Part 1b plan; threat class table is generic and not tied to this crew; audit trail evaluation answers fewer than three of the four questions; top 3 risks and mitigations are vague or missing. | **Points: 21** — Design vs. reality section identifies some changes but does not explain all of them; tool scope table is mostly complete but discrepancies are unexplained; threat class table addresses the crew but mitigations are stated without specificity; audit trail evaluation answers all four questions but one or more stays general (in particular, the human-approval-task question gets a generic answer rather than walking through the actual Timeline entries); risks and mitigations are named but not specific to this crew or not prioritized. | **Points: 30** — Design vs. reality section honestly compares the Copilot's output to the final crew and explains every change; tool scope table is complete and discrepancies are explained; threat class table is specific to this crew with concrete mitigations (not aspirational statements); audit trail evaluation answers all four questions with specifics from this build, including a clear-eyed walk-through of whether the human-approval task actually paused or fabricated approval; top 3 risks and mitigations are crew-specific and ordered by priority. |
| **Reflection (Part 5)** | **Points: 6** — One or both questions missing, or answered with general statements not tied to this build. | **Points: 11** — Both questions answered, but Question 1 describes the governance decision without explaining the reasoning behind it, or Question 2 names a general role ("IT," "the developer") rather than a specific function. | **Points: 15** — Question 1 names a specific governance decision made during the build and explains the judgment behind it (not just what was done, but why); Question 2 names an actual organizational function (not "IT" or "the engineer") and identifies the specific timeline entry that function would need to investigate the failure. |
